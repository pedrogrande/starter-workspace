This document records every issue encountered during the Agent-UI integration into the Coder workspace, with root cause analysis and fixes.

**Date:** 2026-06-22
**Agent-UI fork:** <https://github.com/pedrogrande/agent-ui>
**Workspace image:** `ghcr.io/pedrogrande/course-workspace:latest`

## Issue 1: Missing CA Certificates in Node Builder Stage

### Symptom

```
fatal: unable to access 'https://github.com/pedrogrande/agent-ui.git/': server certificate verification failed. CAfile: none CRLfile: none
```

### Root Cause

The `node:22-slim` base image for the Agent-UI builder stage doesn't include CA certificates. `git clone` over HTTPS requires CA certificates to verify GitHub's TLS certificate.

### Fix

Added `ca-certificates` to the apt-get install in the builder stage:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends git ca-certificates && rm -rf /var/lib/apt/lists/*
```

### Explanation

Slim Docker images strip out packages not essential for the runtime. `ca-certificates` is needed for any HTTPS connection (git clone, curl, npm registry). Always include it when building from slim images that need network access.

## Issue 2: pnpm Lockfile Config Mismatch

### Symptom

```
[ERR_PNPM_LOCKFILE_CONFIG_MISMATCH] Cannot proceed with the frozen installation. The current "overrides" configuration doesn't match the value found in the lockfile
```

### Root Cause

The `pnpm install --frozen-lockfile` flag requires the lockfile to exactly match `package.json`. The fork's `package.json` had `pnpm.overrides` that weren't reflected in the upstream lockfile. The lockfile was generated by the original repo, not the fork.

### Fix

Changed to `pnpm install --no-frozen-lockfile` to allow pnpm to regenerate the lockfile:

```dockerfile
RUN git clone https://github.com/pedrogrande/agent-ui.git . && \
    pnpm install --no-frozen-lockfile
```

### Explanation

`--frozen-lockfile` is for CI/CD where you want to guarantee reproducible installs. When you've modified `package.json` (added `pnpm.overrides`, `pnpm.onlyBuiltDependencies`, `packageManager`), the lockfile must be regenerated. Use `--no-frozen-lockfile` in Docker builds where the repo may have been modified after the lockfile was last generated.

## Issue 3: Sharp Build Script Failure (pnpm 10+)

### Symptom

```
[ERR_PNPM_IGNORED_BUILDS] Ignored build scripts: sharp@0.34.5
Run "pnpm approve-builds" to pick which dependencies should be allowed to run scripts.
```

### Root Cause

`sharp@0.34.5` is a native image processing library with no prebuilt binary for Node 22+. pnpm 10+ auto-detects packages with install scripts and requires explicit approval to run them. `pnpm approve-builds` is interactive — it can't run in a Docker build.

Multiple approaches failed:

- `.npmrc` settings (`ignore-build-scripts`, `onlyBuiltDependencies`) — overridden by pnpm's auto-detection
- `pnpm config set ignore-build-scripts false` — unsupported YAML config key in pnpm 10+
- `pnpm approve-builds` — interactive, can't run non-interactively in Docker

### Fix

Added `pnpm.onlyBuiltDependencies` whitelist to `package.json` in the fork (the **only** mechanism that works with pnpm 10+):

```json
{
  "pnpm": {
    "onlyBuiltDependencies": ["esbuild"]
  }
}
```

This tells pnpm: "only run build scripts for `esbuild`". Everything else (including `sharp`) is implicitly blocked. Also pinned the pnpm version:

```json
{
  "packageManager": "pnpm@10.33.0"
}
```

### Explanation

See `docs/sharp-build-fix.md` (referenced in `/memories/repo/sharp-build-fix.md`) for the full analysis. The key takeaway: **`pnpm.onlyBuiltDependencies` in `package.json`** is the canonical way to control build scripts in pnpm 10+. pnpm 11 changed to `allowBuilds` — pin the version to avoid conflicts.

## Issue 4: ESLint `no-explicit-any` Errors During Next.js Build

### Symptom

```
Error: Unexpected any. Specify a different type.  @typescript-eslint/no-explicit-any
```

Next.js build failed because `next build` runs ESLint, and the custom view code used `Record<string, any>` types.

### Root Cause

The Agent-UI project has `@typescript-eslint/no-explicit-any` enabled as an error (not warning). The TypeScript interfaces I added for the AgentOS API used `Record<string, any>` for flexible JSON fields (session_state, metrics, metadata, etc.).

### Fix

Replaced all `Record<string, any>` with `Record<string, unknown>` in `src/types/os.ts`:

```bash
sed -i '' 's/Record<string, any>/Record<string, unknown>/g' src/types/os.ts
```

Also fixed a specific `any` in `SessionsView.tsx` where chat history entries were typed as `any`.

### Explanation

`unknown` is the type-safe alternative to `any` — it forces the consumer to narrow the type before using it. For API response types where the structure is flexible (e.g., `session_state` can contain anything), `Record<string, unknown>` is the correct choice. It satisfies ESLint while still being flexible.

## Issue 5: Wrong Export Name for Textarea Component

### Symptom

```
Type error: '"@/components/ui/textarea"' has no exported member named 'Textarea'. Did you mean 'TextArea'?
```

### Root Cause

The Agent-UI project exports the textarea component as `TextArea` (capital A in "Area"), but my view components imported it as `Textarea` (lowercase a). This is a case sensitivity mismatch — TypeScript treats it as a type error.

### Fix

Changed imports in `MemoryView.tsx` and `KnowledgeView.tsx`:

```typescript
// Wrong
import { Textarea } from '@/components/ui/textarea'
<Textarea ... />

// Correct
import { TextArea } from '@/components/ui/textarea'
<TextArea ... />
```

### Explanation

Always check the actual export name before importing. The shadcn/ui convention is inconsistent — some projects use `Textarea`, others use `TextArea`. A quick `grep "export" src/components/ui/textarea.tsx` reveals the correct name.

## Issue 6: Docker Build Cache Serving Stale Agent-UI Code

### Symptom

After pushing fixes to the agent-ui fork, the Docker build still used the old code with the `Textarea` import error. Even `--no-cache` didn't help.

### Root Cause

Docker buildx maintains its own cache separate from the regular Docker cache. The `--no-cache` flag skips the regular cache but buildx may still use cached layers from previous builds. The `git clone` step appeared to run (not CACHED), but the buildx cache was serving stale content from a previous layer.

### Fix

Cleared the buildx cache entirely:

```bash
docker buildx prune -f
```

This removed 21.3GB of cached layers. The subsequent build cloned the latest code from GitHub and built successfully.

### Explanation

Docker buildx uses a separate cache store from the regular Docker daemon. `--no-cache` tells buildx not to use the cache for this build, but there can be edge cases where cached content persists. `docker buildx prune -f` clears all cached layers, forcing a completely fresh build. Use this when you've pushed changes to a git repo that's cloned during the build and the cache seems stale.

## Issue 7: Path-Based Proxy Breaking Next.js Static Assets

### Symptom

```
Refused to apply style from 'http://74.208.237.168/_next/static/css/31d31879efc140bd.css' because its MIME type ('text/html') is not a supported stylesheet MIME type, and strict MIME checking is enabled.
```

### Root Cause

With `subdomain = false` (the default for `coder_app`), Coder proxies the app via a path-based URL like `http://74.208.237.168/@pedrog/starter-one/apps/agent-ui/`. Next.js generates absolute asset paths like `/_next/static/css/...` in its HTML. The browser resolves these to `http://74.208.237.168/_next/static/css/...` — missing the Coder proxy subpath. The Coder server returns HTML (the dashboard) for unknown paths, causing the MIME type mismatch.

### Fix

1. Set up a domain with wildcard DNS: `*.coder.viberiders.club` → `74.208.237.168`
2. Configured `CODER_WILDCARD_ACCESS_URL=*.coder.viberiders.club` on the VPS
3. Set `subdomain = true` on both `coder_app` resources in the template

With subdomain proxying, the app is accessed at `agent-ui--starter-one--pedrog.coder.viberiders.club` — its own origin, so absolute asset paths work correctly.

### Explanation

The Coder docs state: "`CODER_WILDCARD_ACCESS_URL` is necessary for port forwarding via the dashboard or running `coder_apps` on an absolute path." Path-based proxying (`subdomain = false`) doesn't work with SPAs like Next.js that generate absolute asset paths. Subdomain proxying (`subdomain = true`) gives each app its own origin, avoiding the path mismatch.

**Key lesson:** Always use `subdomain = true` for Next.js (or any SPA) apps in Coder. This requires a domain with wildcard DNS — plan for this during infrastructure setup.

## Issue 8: Default Endpoint Showing Port 7777 Instead of 8000

### Symptom

The Agent-UI sidebar showed `http://localhost:7777` as the default endpoint, but AgentOS runs on port 8000.

### Root Cause

The upstream Agent-UI repo defaults to `http://localhost:7777` (Agno's default dev port). The store change to `http://localhost:8000` was committed to the fork but the Docker build cached the old code.

### Fix

Changed the default in `src/store.ts`:

```typescript
selectedEndpoint: 'http://localhost:8000',
```

Combined with the buildx cache clear (Issue 6), the fix was picked up in the next build.

### Explanation

The default endpoint is persisted in `localStorage` via Zustand's `persist` middleware. If a student previously connected to a different endpoint, the persisted value takes precedence over the default. The default only applies on first visit or when `localStorage` is cleared.
